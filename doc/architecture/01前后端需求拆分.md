# 视频编辑器前后端需求拆分

## 1. 文档目的

本文档用于将当前视频编辑器需求拆分为前端职责与后端职责，方便后续分别推进 UI、交互、Python 工作流与接口设计。

## 2. 总体原则

- 前端负责配置、展示、交互、状态反馈和结果确认
- 后端负责本地能力调用、模型管理、任务执行、数据落盘和统一接口输出
- 当前模型接入以外部 API 服务为主，前端负责统一配置与展示，后端负责封装调用差异

## 3. 前端需求

### 3.1 编辑器主界面

- 提供视频编辑器主界面
- 至少包含素材区、预览区、时间轴区、属性区
- 承载自动剪辑结果展示与人工微调

### 3.2 素材导入与识别界面

- 提供素材导入入口
- 展示素材列表与基础状态
- 提供“开始识别 / 重新识别”入口
- 展示识别结果摘要，例如标签、片段建议、字幕状态

补充说明：

- 参考现有 `VideoSceneAnalyzer`，素材识别页建议直接支持批量视频列表
- 需要支持拖拽视频、移除选中、清空列表
- 需要支持分析结果预览，包括整体摘要、音频转写文本和关键帧缩略图

### 3.3 配音 / 旁白界面

- 提供录音入口
- 提供音频导入入口
- 提供文字转语音输入区域
- 展示生成结果、试听入口和插入时间轴入口

补充说明：

- 参考现有 `mlx-whisper`，配音 / 旁白界面还应支持：
  - 现有音色选择
  - 上传样本创建音色
  - 录音创建音色
  - 试听 TTS 结果
  - 将 TTS 结果直接插入到选中片段之后

新增目标：

- 支持从当前项目已有片段中直接选择一个片段作为音色样本

### 3.4 智能剪辑界面

- 提供多种剪辑工具入口
- 支持按 AI 标签剪辑
- 支持按音频 SRT 剪辑
- 支持按口头语关键词剪辑
- 支持删除冗余片段和重复片段
- 展示剪辑候选结果，并允许用户确认或调整

补充说明：

- 一个项目内会有多个视频片段和多个音轨片段
- 导入视频时需要将视频轨和音频轨分离
- 如果具体剪辑交互没有单独说明，默认参考 `Final Cut Pro`

### 3.5 导出界面

- 提供导出参数配置
- 支持导出预设保存与选择
- 支持导出目录选择
- 展示导出任务状态

### 3.6 模型管理界面

- 提供独立模型页面
- 按模型类型分类展示
- 支持 API 模式配置
- 支持填写 API 地址和 key
- 支持显示模型配置状态
- 支持帮助文档入口

当前明确的模型分类：

- VL 模型
- LLM 模型
- 音频处理模型
- 声音合成模型

补充说明：

- 当前不考虑在应用内下载模型
- 模型页以外部服务接入为主
- 参考现有 `VideoSceneAnalyzer`，模型页至少要覆盖图片模型和音频模型的当前选中状态

### 3.7 依赖管理界面

- 提供依赖检测 UI
- 展示 ffmpeg 等关键依赖是否已安装
- 展示 Python 包是否完整
- 提供安装按钮
- 展示安装进度、结果和失败信息

补充说明：

- 参考现有 `VideoSceneAnalyzer`，依赖页或依赖区域建议支持：
  - 安装依赖
  - 启动服务
  - 停止服务
  - 健康状态显示
  - 安装 / 启动日志展示

### 3.8 AI 工作流管理界面

- 提供工作流列表
- 支持启用与停用
- 支持顺序调整
- 支持编辑 prompt 模板
- 支持展示最近一次测试或运行结果
- 支持展示 AI 返回的结构化操作建议

### 3.9 音色管理界面

- 提供独立音色库或可嵌入式音色选择器
- 支持上传参考音频创建音色
- 支持录音创建音色
- 支持试听参考样本
- 支持删除音色
- 支持设置默认音色
- 支持从现有项目片段提取为音色样本

### 3.10 前端工程结构约束

- 前端全局状态优先使用 `zustand`
- 局部临时 UI 状态保留在组件内部
- 基础 UI 组件与业务组件分层
- `Radix` / `shadcn` 基础组件统一放在 `components/ui`
- 业务模块按 `features` 组织
- store 统一放在 `stores`
- 数据请求与适配逻辑放在 `lib` 或 `services`

补充说明：

- 当前不建议一开始引入更重的全局状态方案
- 编辑器、时间轴、模型页、项目会话这类跨区域共享状态，适合放在 `zustand`
- 展示组件应尽量只接收数据和事件，不直接承担复杂请求逻辑

### 3.11 前端技术基线

- 视频剪辑相关前端能力以 `Remotion` 为核心
- 前端框架基线使用 `Next.js 16`
- 基础组件优先使用官方 `shadcn/ui`
- 不额外重做一套与 `shadcn/ui` 重复的基础 Tailwind UI
- 若可用，开发前优先接入 MCP 辅助能力

补充说明：

- 根据 Next.js 官方发布与支持策略，`16.x` 于 `2025-10-21` 发布，当前为 Active LTS
- 因此前端以 `Next.js 16` 作为基线版本是合理的
- `shadcn/ui` 与 `Radix` 仍统一归入 `components/ui`
- 所有视频编辑相关前端能力统一由 `Remotion` 承担
- 包括时间轴、切片、预览、导出等
- 不额外并行设计第二套视频编辑表达体系

## 4. 后端需求

### 4.1 项目与数据管理

- 管理项目创建、打开、保存
- 管理素材元信息
- 管理时间轴数据
- 管理导出配置与预设
- 管理模型配置与依赖状态数据

补充说明：

- 每个项目以一个项目文件夹为核心
- 项目数据以项目目录内的 `project.json` 为主入口
- 项目目录第一版建议至少包含：
  - `project.json`
  - `videos/`
  - `frames/`
  - `audio/`
  - `captions/`
- 一个项目内需要支持多个视频片段与多个音轨片段
- 导入视频时需要拆分为视频轨和音频轨
- 原视频需要支持两种模式：
  - 复制到项目目录
  - 不复制，只记录外部原路径
- 应用初期启动时需要支持：
  - 打开项目文件夹
  - 新建项目
- 后端需要负责初始化项目目录结构，并负责读写 `project.json`

### 4.2 素材处理工作流

- 接收素材导入请求
- 执行媒体信息解析
- 调用 VL 模型或外部 API 对视频抽帧分析
- 输出固定 JSON 结果
- 将结构化结果提供给 LLM 工作流使用

补充说明：

- 参考现有 `VideoSceneAnalyzer`，这一工作流已经可以进一步细化为：
  - `ffprobe` 获取视频时长
  - `scenedetect` 检测镜头切换
  - 或按固定间隔生成截图时间点
  - `ffmpeg` 提取关键帧
  - 调用图片模型生成结构化描述
  - 解析成 `shot_type / subject / action / environment / other`
  - 输出到单视频 JSON 与汇总 JSON

### 4.3 LLM 剪辑工作流

- 接收 VL 输出结果
- 按统一协议调用本地或远程 LLM
- 输出剪辑建议、标签或剪辑动作
- 将结果转换为前端可直接消费的剪辑候选数据

补充说明：

- 参考现有 Python 实现，LLM 工作流已在“字幕片段分析”链路上得到验证
- 当前已验证模式是：将带时间戳的 segments 提交给 LLM，返回严格 JSON
- 该 JSON 可包含删除片段、删除片段内文本、打标签、片头候选等操作
- 视频分析阶段可直接复用这套“结构化输入输出协议 + 操作回写”模式

### 4.4 音频转写工作流

- 管理音频处理模型配置
- 调用 API 模式的音频处理能力
- 输出 SRT 或相关转写结果
- 为字幕剪辑和口头语检索提供数据

补充说明：

- 参考现有 `VideoSceneAnalyzer`，音频转写工作流已验证以下链路：
- 从视频中抽取 `16k` 单声道 `wav`
- 调用本地 `FunASR` 服务
- 或调用 `oMLX` OpenAI 兼容音频接口
- 返回 transcript 与可选 segments

补充说明：

- 参考现有 `mlx-whisper`，音频转写工作流还应支持：
  - 对整段导入音频执行转录
  - 直接导入 SRT 解析为 segments
  - 对单独一个插入片段再次转录
  - 生成词级时间戳 `words`

### 4.5 声音合成工作流

- 管理 TTS 模型配置
- 调用 API 模式的声音合成能力
- 根据文本生成音频
- 将生成结果写入项目资源目录并返回元信息

补充说明：

- 参考现有 `mlx-whisper`，声音合成工作流已验证以下链路：
  - 音色样本上传 / 录音
  - 音色元信息保存
  - 使用 `IndexTTS-2` 生成试听与正式语音
  - 返回异步 job_id
  - 轮询生成状态
  - 完成后将音频文件写入项目目录
  - 记录每段 TTS 的语速、语气、首尾静音等参数

新增目标：

- 支持从当前项目已有片段中提取参考音频，作为音色样本来源

### 4.5.1 语音片段插入工作流

- 支持录音片段插入
- 支持 TTS 片段插入
- 支持插入到指定片段之后
- 插入后顺延后续片段时间
- 尝试将新片段拼接进主音频
- 支持对插入后的新片段再次转文字

### 4.6 模型管理

- 管理模型分类
- 保存 API 模式配置
- 提供模型帮助文档
- 统一返回模型状态

补充说明：

- 参考现有 `VideoSceneAnalyzer`，至少已经验证了以下模型接入形态：
  - 远程 `MiniMax` 图片分析
  - 本地 `oMLX` 视觉模型
  - 本地 `FunASR` 音频模型
  - 本地 `oMLX` 音频模型

### 4.7 依赖检测与安装

- 检测 ffmpeg、ffprobe 和关键 Python 包
- 返回依赖状态、版本和错误信息
- 提供依赖安装入口
- 记录安装日志与安装结果

补充说明：

- 参考现有 `VideoSceneAnalyzer`，本地依赖管理已经验证了“安装依赖 -> 启动服务 -> 健康检查 -> 停止服务”的模式
- 这一模式可直接推广到后续 VL、Whisper、TTS、其他本地模型和基础依赖的安装管理

### 4.9 存储与索引管理

- 文件系统负责项目实体数据
- SQLite 负责应用级设置和索引数据
- 维护最近打开项目记录
- 维护模型配置与依赖状态
- 维护全局音色索引
- 在需要时缓存便于查询的数据，但不替代项目文件本身

### 4.8 导出工作流

- 根据时间轴和导出配置执行导出任务
- 支持预设读取与保存
- 支持导出目录写入
- 返回导出任务状态与结果路径

## 7. 当前仍待补充的问题

- VL 固定 JSON 的最终字段定义
- LLM 返回的剪辑动作格式
- 依赖安装在不同操作系统上的差异
- API 模式是否支持多个配置方案切换
- 当前字幕分析工作流如何平滑扩展到视频 VL 分析工作流
- 哪些字段可以在音频分析和视频分析之间共用
- 当前视频分析 JSON 如何映射为后续剪辑引擎可消费的片段和标签
- “从现有片段创建音色”是否自动生成独立 voice_id
- 音色库是全局级还是项目级
- 插入后的 TTS 片段是否默认自动做单片段转录
- `project.json` 中 timeline、media_items、analysis_results 的最终字段结构
- SQLite 中哪些数据是权威数据，哪些只是索引或缓存
- 多视频片段与多音轨在时间轴中的最终数据结构
- 模型页是否只做配置与文档，还是保留最小连通性检查

## 8. 当前结论

当前新增需求已经明确，这个应用除了编辑器本体之外，还必须具备两个基础能力层：

- 模型管理层：负责 API 地址 + key 配置、帮助文档和状态管理
- 环境依赖层：负责 ffmpeg、ffprobe、Python 包等运行条件的检测与安装

这两层能力应作为前端设置体系和后端基础设施体系的一部分优先设计。

补充结论：

- 参考现有 `VideoSceneAnalyzer`，已经可以确认“视频镜头检测/定时截图 -> VL 画面分析 -> 音频抽取与转写 -> 单视频 JSON -> 批量汇总 JSON”的链路是可行的
- 当前新增工作不再是从零设计视频分析，而是把这套分析结果进一步转化为剪辑标签、剪辑建议和时间轴动作
- 因此后续设计时应优先复用已有的视频分析结构、依赖管理方式和服务状态管理方式
