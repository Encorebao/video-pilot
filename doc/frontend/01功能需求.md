# 视频编辑器功能需求

## 1. 文档目的

本文档用于持续整理当前视频编辑器的产品需求，作为后续前端设计、交互梳理、后端接口设计与联调的依据。

当前阶段目标：

- 明确产品核心功能边界
- 将需求按模块拆分，便于后续逐步实现
- 保留待补充项，方便后续持续增量更新

## 2. 产品定位

这是一个本地视频编辑器，重点不是传统手工剪辑，而是围绕 AI 识别、音频处理、字幕处理和自动剪辑能力来提升剪辑效率。

核心特点：

- 支持素材导入与识别分析
- 支持录音、音频导入和文字转配音
- 支持基于 AI 标签、字幕、口头语等条件进行自动剪辑
- 支持导出参数配置、预设保存和导出目录管理

## 3. 当前确定的核心功能

### 3.1 素材导入与处理

目标：让用户能够将视频素材导入编辑器，并通过识别能力为后续剪辑生成可用数据。

当前已明确需求：

- 支持导入素材
- 支持对素材进行处理
- 这里的“处理”当前指 AI / API 识别能力

处理结果应服务于后续剪辑，至少包括：

- 为视频生成可用于剪辑的标签
- 为后续自动剪辑提供结构化识别结果

当前暂不展开的细节，后续补充：

- 支持导入的素材类型范围
- 单次导入数量限制
- 识别结果的具体字段结构
- 识别由本地模型完成还是外部 API 完成

### 3.2 配音 / 旁白

目标：让用户可以围绕视频内容快速生成或导入旁白音频。

当前已明确需求：

- 支持录音
- 支持导入音频
- 支持通过文字生成音频

该模块应覆盖的输入方式：

- 实时录音
- 本地音频文件导入
- 文本转语音生成

该模块后续需要进一步明确的内容：

- 是否支持多段旁白管理
- 是否支持重新生成与版本保留
- 文字转语音的音色、语速、情绪等参数范围
- 录音后的降噪、裁剪、试听是否需要内置

补充明确需求：

- 在语音插入片段场景中，除录音外，还需要支持 TTS
- 用户可以选择一个样本片段作为音色参考
- 用户输入文字后，生成一个新的音频片段
- 新生成的音频片段可以插入到时间轴指定位置

### 3.3 视频剪辑

目标：根据 AI 识别结果、字幕内容和音频文本信息，对视频进行自动化剪辑。

当前已明确需求：

- 根据 AI 生成的视频标签进行剪辑
- 根据音频 SRT 进行剪辑
- 删除 AI 建议的多余片段
- 删除重复片段
- 根据特定字幕内容进行剪辑

当前已明确的特定字幕剪辑场景：

- 搜索并剪辑“嗯”
- 搜索并剪辑“啊”
- 搜索并剪辑“额”

从功能角度，该模块当前可拆为以下剪辑工具：

- 基于 AI 标签的剪辑工具
- 基于 SRT 的剪辑工具
- 基于 AI 建议结果的冗余片段删除工具
- 基于重复内容识别的删除工具
- 基于关键词字幕检索的剪辑工具

后续还需要继续明确：

- 自动剪辑后是否允许逐条确认
- 自动剪辑结果是否支持回退
- “重复片段”的判断依据
- “多余片段”的识别标准
- SRT 的来源是外部导入、音频转录生成，还是两者都支持

补充明确需求：

- 一个项目内会有多个视频片段
- 也会有多个音轨片段
- 导入视频时，视频轨和音频轨需要分离
- 整体视频编辑交互参考 `Final Cut Pro`
- 如果某些视频剪辑细节没有单独说明，默认参考 `Final Cut Pro`

### 3.4 导出

目标：让用户能够按固定规格输出成片，并重复使用常用导出配置。

当前已明确需求：

- 支持设置导出规格
- 导出规格可保存为预设
- 支持设置导出目录

导出模块当前至少需要包含：

- 导出参数设置
- 预设管理
- 导出目录设置

后续需要补充的内容：

- 支持哪些导出格式
- 是否支持分辨率、码率、帧率等详细参数
- 是否支持导出音视频分离文件
- 是否支持批量导出

### 3.5 模型与依赖管理

目标：让用户在应用内完成模型 API 配置、帮助文档查看和依赖确认，确保 AI 剪辑相关能力可直接运行。

当前已明确需求：

- UI 中需要有独立的模型页面
- 模型页面改为配置 API 地址和 key
- 暂时不考虑在 app 内下载模型
- 每类模型提供帮助文档入口
- 应用还需要提供依赖安装与依赖状态判断的 UI
- 依赖中至少包括 ffmpeg 以及相关 Python 包

当前已明确的模型分类：

- VL 模型
- LLM 模型
- 音频处理模型
- 声音合成模型

#### 3.5.1 VL 模型

用途：

- 对视频进行抽帧
- 对抽取出的画面进行分析
- 输出固定 JSON
- 该 JSON 提供给后续 LLM 模型做剪辑决策

当前明确要求：

- 采用 API 接入形式
- 需要配置 API 地址和 key
- 需要提供帮助文档入口
- 输出结果必须是固定 JSON 结构

后续需要继续明确：

- 抽帧策略是固定间隔、镜头变化抽帧，还是两者都支持
- 固定 JSON 的字段定义
- 接入文档中的请求示例与最小必填参数

#### 3.5.2 LLM 模型

用途：

- 接收 VL 输出结果
- 结合业务规则生成剪辑建议、剪辑标签或剪辑动作

当前明确要求：

- 采用 API 接入形式
- 需要配置 API 地址和 key
- 需要提供帮助文档入口

后续需要继续明确：

- 是否允许多个 LLM 提供商并存
- 输入输出协议是否统一
- 剪辑建议的结果结构定义

#### 3.5.3 音频处理模型

用途：

- 将音频转换为 SRT

当前明确要求：

- 采用 API 接入形式
- 需要配置 API 地址和 key
- 需要提供帮助文档入口

后续需要继续明确：

- 是否只输出 SRT，还是同时输出逐词时间戳
- 是否支持多语言
- 是否支持说话人分离

补充明确需求：

- 音频处理能力不仅用于独立音频文件，也用于从视频中抽取出的音频
- 对于后续插入的录音片段、TTS 片段，也最好支持单片段再次转文字

#### 3.5.4 声音合成模型

用途：

- 通过文字生成语音
- 当前示例模型类型为 `index-tts` 一类的声音合成能力

当前明确要求：

- 采用 API 接入形式
- 需要配置 API 地址和 key
- 需要提供帮助文档入口

后续需要继续明确：

- 是否支持音色列表管理
- 是否支持试听
- 是否支持语速、停顿、情绪参数

补充明确需求：

- 需要支持音色克隆或音色参考能力
- 音色参考来源至少应包括：
  - 上传参考音频
  - 录音生成参考音频
  - 从现有项目片段中选择一个片段作为音色样本
- TTS 生成后的结果需要能够直接形成一个可插入时间轴的音频片段
- TTS 当前按“文字转语音后插入音频片段”理解

#### 3.5.5 依赖检测与安装

目标：

- 让用户在进入应用后可以明确知道本地运行环境是否完整
- 缺失依赖时可在 UI 中触发安装或查看安装状态

当前已明确需求：

- 需要有依赖安装判断的 UI 页面或区域
- 需要检测 ffmpeg 等关键依赖
- 需要检测相关 Python 包

从产品层面建议该模块至少包含：

- 依赖检测状态
- 已安装 / 未安装 / 版本异常状态展示
- 安装入口
- 安装结果反馈
- 失败重试能力

后续需要继续明确：

- 依赖安装是自动执行还是仅提供指引
- Windows / macOS 的安装流程是否分别处理
- Python 环境由应用托管还是用户自行准备

#### 3.5.6 模型帮助文档

当前明确需求：

- 每类模型都应提供帮助文档或接入说明
- 文档用于告诉用户如何准备外部服务
- 文档至少应说明：
  - API 地址如何填写
  - key 如何填写
  - 该模型的用途
  - 推荐请求方式或服务要求

示例：

- TTS 模型页可提供 TTS 服务接入文档
- 用户只需要填写 TTS 的 API 地址和 key

### 3.6 参考现有 Python 实现提炼出的已验证能力

说明：

- 已参考你现有的 Python 应用 `VideoSceneAnalyzer`
- 该应用已经验证了“视频镜头检测 / 定时截图 / VL 画面分析 / 音频转写 / JSON 报告输出”的一条完整链路
- 因此这些能力可以直接写入当前产品需求，作为后续视频编辑器的视频分析基线

#### 3.6.1 已验证的视频分析主流程

当前参考实现已经具备以下流程：

1. 导入一个或多个视频文件
2. 获取视频时长
3. 选择分析模式：
   - 镜头检测模式
   - 定时截图模式
4. 提取关键帧图片
5. 调用图片模型对关键帧做结构化描述
6. 从视频中抽取音频为 `16k` 单声道 `wav`
7. 调用音频模型执行转写
8. 输出单视频 JSON 报告
9. 输出批量汇总 `summary.json`

这说明当前产品中的“素材导入与处理”模块，不能只停留在抽象描述，而应明确包含：

- 视频时长解析
- 镜头切分或定时采样
- 关键帧提取
- 画面结构化描述
- 音频抽取与转写
- 单视频结果与批处理结果保存

#### 3.6.2 已验证的 VL / 视频分析能力

参考实现中已经落地的 VL 相关能力包括：

- 基于 `scenedetect` 自动检测镜头切换点
- 基于 `ffmpeg` 在指定时间点提取关键帧
- 支持两种抽帧策略：
  - 根据镜头切换点取每段中间帧
  - 按固定秒数间隔截图
- 支持通过图片模型对关键帧做结构化文字描述
- 当前图片模型通道至少包括：
  - `MiniMax` 远程视觉接口
  - 本地 `oMLX` 兼容视觉模型

当前参考实现中的画面描述目标格式是：

- `[景别]`
- `[主体]`
- `[动作]`
- `[环境]`
- `[其他]`

这说明你当前产品中的 VL 模型需求可以进一步落地为：

- 不只是“输出固定 JSON”
- 而是先输出稳定的画面描述字段，再由后续 LLM 或剪辑规则继续消费

#### 3.6.3 已验证的音频分析能力

参考实现中已经落地的音频处理能力包括：

- 直接从视频中抽离音频
- 音频统一转换为 `16k`、单声道 `wav`
- 支持两类音频转写方式：
  - 本地 `FunASR`
  - `oMLX` OpenAI 兼容音频接口
- 转写结果写入 JSON
- 支持返回 transcript 文本
- 在部分模型模式下支持返回 segments

这意味着你当前定义的“音频处理模型”需求，建议明确为：

- 输入是音频文件，也可以来自视频抽取音频
- 输出至少包含 transcript
- 如模型支持，输出 segments / 时间戳结构
- 结果要能直接给字幕剪辑与检索剪辑使用

#### 3.6.4 已验证的输出 JSON 结构

参考实现中，单视频结果已经接近一个可直接复用的固定结构，至少包括：

- `video`
- `video_path`
- `output_dir`
- `video_duration`
- `analyzed_at`
- `image_model`
- `audio_model`
- `visual_analysis`
- `audio_analysis`
- `overall_summary`
- `analysis_time_seconds`
- `analysis_time_str`

其中 `visual_analysis` 至少包括：

- `model`
- `total_scenes`
- `scenes`

其中每个 `scene` 至少包括：

- `index`
- `start`
- `end`
- `duration`
- `raw_description`
- `shot_type`
- `subject`
- `action`
- `environment`
- `other`
- `keyframe`

其中 `audio_analysis` 至少包括：

- `enabled`
- `model`
- `audio_file`
- `transcript`
- `segments`
- `error`

这部分非常关键，因为它已经可以直接作为你后续“视频分析固定 JSON”的第一版候选协议。

#### 3.6.5 已验证的 GUI 交互模式

参考实现中已经验证了以下 GUI 交互方式：

- 支持拖拽任意本地视频文件
- 支持批量视频列表管理
- 支持选择输出目录
- 支持设置截图间隔
- 支持显示当前图片模型与音频模型
- 支持本地依赖安装按钮
- 支持本地服务启动 / 停止
- 支持运行日志显示
- 支持分析进度条
- 支持结果预览：
  - 整体摘要
  - 音频转写文本
  - 关键帧缩略图

这说明你的视频编辑器在前端设计上，至少应该把“模型页”和“素材分析页”拆开：

- 模型页负责模型、依赖、服务状态
- 素材分析页负责视频列表、分析参数、运行过程和结果预览

#### 3.6.6 已验证的依赖与本地服务模式

参考实现中已经验证了以下本地依赖 / 服务模式：

- 系统依赖 `ffmpeg`
- 视觉模型可通过本地 `oMLX` 服务接入
- 音频模型可通过本地 `FunASR` 服务接入
- GUI 内可触发 `FunASR` 依赖安装
- GUI 内可启动和停止 `FunASR` 服务
- 启动后通过本地端口健康检查确认服务是否可用
- 首次启动时允许自动下载模型

这意味着你之前提的“模型页面”和“依赖安装判断 UI”是完全合理的，而且已经有参考交互原型。

#### 3.6.7 当前仍需要补充的部分

虽然 `VideoSceneAnalyzer` 已经实现了视频分析链路，但仍有一些内容需要在本项目里继续补强：

- 将当前视频分析 JSON 转换为可直接驱动剪辑的标签协议
- 增加重复片段识别与冗余片段识别
- 将画面分析与字幕分析统一到同一套剪辑决策层
- 将分析结果真正写入时间轴而不只是输出报告
- 增加模型配置保存、帮助文档展示和可选的最小连通性检查 UI

### 3.7 参考现有 Python 实现提炼出的语音转文字与 TTS 能力

说明：

- 已参考你现有的 Python 应用 `mlx-whisper`
- 该项目已经验证了“音频转文字”“音色管理”“文字转语音”“把生成结果插入片段流”的一批能力
- 这些能力可以直接写入当前视频编辑器需求，尤其适合作为语音片段插入和配音旁白模块的基础

#### 3.7.1 已验证的语音转文字能力

当前参考实现已经支持：

- 上传音频后使用 `mlx_whisper` 转录
- 直接导入 SRT 并解析为片段
- 输出 segments 列表
- 输出每段的 `start / end / duration / text`
- 在可用情况下输出逐词时间戳 `words`
- 支持把转录结果写回项目 metadata
- 支持生成 SRT 文件
- 支持对单独一个片段再次转录

从产品角度，这意味着当前项目中的语音转文字需求应明确包含：

- 整段素材的转录
- 插入后单片段的补转录
- transcript 与 words 的结构化保存
- SRT 导入与 SRT 导出

#### 3.7.2 已验证的 TTS 与音色能力

当前参考实现已经支持：

- 独立音色库
- 上传参考音频创建音色
- 录音创建音色
- 音色删除
- 试听音色原始样本
- 基于所选音色输入文本生成试听语音
- 正式生成 TTS 音频
- 生成状态轮询
- 生成结果播放
- 按段批量生成 TTS
- 保存默认音色

当前参考实现中的 TTS 参数能力至少包括：

- `voice_id`
- `text`
- `emo_text`
- `speed`
- `lead_silence`
- `tail_silence`

这说明你的视频编辑器中的 TTS 模块，不应只是一个简单“输入文字 -> 出音频”的按钮，而应包含：

- 音色库
- 音色样本管理
- 试听
- 语速控制
- 语气 / 情绪提示词
- 首尾静音控制
- 单段生成与批量生成

#### 3.7.3 已验证的语音片段插入能力

当前参考实现已经支持：

- 在音频工作台中插入新的录音片段
- 在音频工作台中插入新的 TTS 片段
- 选择插入到某个片段之后，或直接追加到末尾
- 插入后自动顺延后续片段时间
- 插入后尝试把新片段拼接到主音频中
- 插入后的新片段保留 `source_type` 和 `source_file`
- 新插入片段可以再做转录，补齐文本和逐词时间戳

这意味着你现在想要的“语音插入片段”功能，其实可以明确拆成：

- 录音插入
- TTS 插入
- 插入位置选择
- 插入后的时间轴顺延
- 插入片段的文本补录 / 自动转录

#### 3.7.4 你当前新增明确的目标需求

在参考实现基础上，你这次又补充了一个更关键的目标：

- 不仅能上传音频或录音来做音色
- 还要能直接选择一个已有片段作为音色样本
- 选择后输入文字
- 输出新的音频片段
- 并将该音频片段插入到项目时间轴中

这部分需求需要明确区分：

- 参考项目 `mlx-whisper` 已经实现了：
  - 上传样本做音色
  - 录音做音色
  - 用选中的音色做 TTS
  - 把 TTS 片段插入时间线
- 当前视频编辑器还需要新增实现：
  - 从已有项目片段中直接提取样本，生成或绑定为音色参考

#### 3.7.5 建议的语音片段插入完整流程

建议将该功能明确为以下用户流程：

1. 用户在时间轴中选中一个片段或选择插入位置
2. 用户选择插入方式：
   - 录音
   - 使用现有音色做 TTS
   - 选择某个已有片段作为音色样本再做 TTS
3. 若为 TTS：
   - 选择音色或创建音色
   - 输入文本
   - 设置语速、语气、停顿等参数
   - 先试听
   - 确认生成
4. 生成音频后插入到目标位置
5. 更新后续片段时间
6. 需要时对该新片段自动转文字，补齐字幕与词级时间戳

也就是说，当前可以直接复用的是“结构化 AI 工作流模式”和“模型/依赖管理模式”，而不是现成的视频 VL 分析能力。

## 5. 当前建议的用户主流程

### 5.1 素材剪辑主流程

1. 创建或打开项目
2. 导入视频素材
3. 导入后将视频轨和音频轨分离
4. 对素材执行识别分析
5. 生成视频标签 / 字幕 / 其他剪辑依据
6. 选择某一种智能剪辑工具
7. 生成剪辑结果
8. 在时间轴中预览和微调
9. 设置导出参数
10. 选择导出目录并导出

### 5.2 旁白生成流程

1. 进入配音 / 旁白模块
2. 选择录音、导入音频或文字转语音
3. 生成或导入旁白音频
4. 将旁白加入项目
5. 结合视频内容进入后续剪辑或导出流程

### 5.3 模型配置与环境准备流程

1. 打开模型或设置页面
2. 查看当前依赖状态
3. 安装缺失依赖，或确认环境已就绪
4. 选择某类模型
5. 查看该模型的接入帮助文档
6. 配置 API 地址和 key
7. 保存配置
8. 将模型投入素材识别、音频转写、配音或剪辑流程

### 5.4 基于参考实现的音频智能剪辑流程

1. 导入音频或 SRT
2. 生成 transcript segments
3. 自动检测口头语和静音候选
4. 将片段送入 AI 工作流
5. AI 返回严格 JSON 剪辑建议
6. 用户选择应用或撤销 AI 操作
7. 导出主字幕、片头字幕或 FCPXML

### 5.5 基于参考实现的视频分析流程

1. 导入一个或多个视频
2. 选择镜头检测模式或定时截图模式
3. 提取关键帧
4. 调用 VL 模型生成结构化画面描述
5. 抽取音频并调用音频模型转写
6. 生成单视频 JSON 报告
7. 生成批量汇总 `summary.json`
8. 将结果提供给后续剪辑模块使用

### 5.6 基于参考实现的语音转文字流程

1. 导入音频或从视频中抽取音频
2. 使用转录模型执行识别
3. 输出 transcript segments
4. 如模型支持，输出逐词时间戳
5. 保存为项目结构化数据
6. 按需导出或生成 SRT

### 5.7 基于参考实现的 TTS 片段插入流程

1. 进入语音插入区域
2. 选择录音或 TTS
3. 若为 TTS，选择音色来源：
   - 现有音色
   - 上传样本创建新音色
   - 录音创建新音色
   - 从现有片段生成音色样本
4. 输入文本并设置参数
5. 试听
6. 确认生成
7. 将生成结果插入到指定片段后或末尾
8. 更新后续片段时间轴
9. 按需对新片段执行转文字

### 5.8 项目打开 / 新建流程

1. 启动应用
2. 选择“打开项目文件夹”或“新建项目”
3. 若为打开项目，选择已有项目目录
4. 读取 `project.json`
5. 恢复素材、分析结果、字幕、时间轴和设置状态
6. 若为新建项目，初始化项目目录和基础文件

### 3.8 项目目录与数据存储

目标：明确项目目录、项目文件、产物目录，以及 SQLite 在应用中的职责边界。

#### 3.8.1 项目目录结构

当前明确需求：

- 每个项目以一个项目文件夹为核心
- 项目文件夹内保存该项目产生的各种数据
- 第一版项目结构可以按如下组织：

```text
project/
  project.json
  videos/
  frames/
  audio/
  captions/
```

其中当前建议含义为：

- `project.json`
  - 项目主文件
  - 保存项目元信息、素材索引、时间轴状态、分析结果索引、导出设置等
- `videos/`
  - 保存多个视频素材或视频片段
- `frames/`
  - 保存抽帧结果、关键帧、缩略图等
- `audio/`
  - 保存从视频分离出的音轨、录音片段、TTS 片段、混音结果等
- `captions/`
  - 保存字幕、SRT、转录结果导出文件等

#### 3.8.2 原视频文件的导入方式

当前明确需求：

- 视频可以导入到项目文件夹中
- 也可以不导入
- 如果不导入，则项目文件夹内可以没有原始视频文件

这意味着当前需要支持两种素材管理方式：

- 导入模式
  - 将原视频复制到项目目录
- 引用模式
  - 不复制原视频，只在项目数据中记录原视频路径

后续需要继续明确：

- 引用模式下，如果原视频被移动、改名或删除，项目如何提示
- 是否允许用户在项目创建时设置默认导入方式
- 是否支持后续在导入模式和引用模式之间切换

#### 3.8.3 SQLite 的职责

当前明确需求：

- 应用会使用 SQLite 管理应用的一些设置或数据

当前建议 SQLite 主要用于：

- 应用级设置
- 最近打开项目记录
- 模型配置
- 依赖安装状态
- 全局音色库索引
- 需要快速查询的索引型数据

当前建议文件系统主要用于：

- 项目主数据
- 素材文件或素材引用信息
- 帧图像
- 音频文件
- 字幕文件
- 导出产物

也就是说，当前需求倾向于：

- 项目内容以项目文件夹和 `project.json` 为主
- SQLite 负责应用级配置和索引，不替代项目文件结构

#### 3.8.4 项目打开与新建方式

当前明确需求：

- 应用初期需要让用户选择：
  - 打开项目文件夹并加载项目
  - 新建项目

这意味着应用启动后至少应有一个项目入口流程：

1. 启动应用

### 3.9 前端工程约束

目标：明确前端开发时的状态管理方式、目录拆分方式和组件分层规则，避免后续页面增长后结构失控。

#### 3.9.1 状态管理

当前建议：

- 桌面端前端状态管理优先使用 `zustand`

原因：

- 足够轻量，适合编辑器类桌面应用
- 学习和维护成本低
- 适合管理时间轴状态、选中状态、面板状态、项目会话状态等全局前端状态
- 比较容易按模块拆 store，避免一开始引入过重方案

当前不建议一开始就引入更重的全局方案，除非后续复杂度明显上升。

建议的状态分层：

- 组件局部状态
  - 仅当前组件内部使用的 UI 状态
  - 例如弹窗开关、输入框内容、hover 状态
- 页面级 / 功能级共享状态
  - 使用 `zustand`
  - 例如当前项目、当前选中素材、当前时间轴选区、当前模型配置、插入面板状态
- 服务端数据状态
  - 保持与 UI 状态分离
  - 不直接混在展示组件中

#### 3.9.2 组件与目录拆分

当前明确需求：

- 前端代码需要按功能模块拆分
- `components` 需要和功能模块结构配合组织
- UI 要和数据逻辑分离
- `Radix` 相关基础组件需要放在独立目录

当前建议的拆分原则：

- `components/ui`
  - 放通用基础 UI 组件
  - 包括 `Radix`、`shadcn/ui` 封装组件
  - 这些组件不直接依赖业务数据
- `components`
  - 放业务展示组件
  - 如编辑器面板、时间轴、属性面板、素材卡片、模型状态卡片等
- `features`
  - 按功能模块组织页面逻辑、状态、数据处理
  - 例如 `project`、`media`、`tts`、`transcript`、`export`
- `stores`
  - 放 `zustand` store
- `lib` 或 `services`
  - 放数据请求、适配器、工具函数
- `types`
  - 放前端使用的数据类型定义

#### 3.9.3 UI 与数据分离

当前明确需求：

- UI 组件和数据逻辑要分离

当前建议规则：

- 基础 UI 组件只负责展示和交互外壳
- 业务组件负责拼装 UI，但尽量不直接写复杂数据请求
- 数据获取、数据转换、状态同步逻辑放到 `features`、`stores`、`lib/services` 中
- 页面组件负责把数据和事件传给展示组件，而不是把所有逻辑堆在页面文件里

简单说，当前前端结构应避免：

- 页面文件里同时堆 UI、状态、请求、转换逻辑
- 通用 UI 组件直接耦合业务字段
- `Radix` 基础组件散落在各业务目录中

#### 3.9.4 前端技术栈约束

当前明确需求：

- 视频剪辑功能使用 `Remotion`
- 前端框架使用 `Next.js 16`
- `shadcn/ui` 优先直接使用官方组件能力
- 不要为了还原基础效果再额外手写一层重复的 Tailwind UI
- 如果有可用的 MCP，先安装或接入，再开始前端开发

补充说明：

- 这里的 `Next.js 16` 指官方已发布的 `16.x` 主版本
- 根据 Next.js 官方发布与支持策略，`16.x` 于 `2025-10-21` 发布，当前属于 Active LTS
- 因此前端基线版本直接定为 `Next.js 16` 是合理的

对于 `shadcn/ui` 的使用约束，当前建议理解为：

- 能用官方现成组件实现的，不重复造基础按钮、弹层、表单、下拉等
- `Radix` / `shadcn` 基础组件统一收敛在 `components/ui`
- 页面和业务组件可以组合这些组件，但不要重新手搓一套外观相近的基础库

对于 `Remotion` 的使用约束，当前建议理解为：

- 所有视频编辑相关能力统一使用 `Remotion`
- 包括但不限于：
  - 时间轴表达
  - 片段切片
  - 预览
  - 导出
- 不再另外发散成第二套视频编辑实现逻辑
- 也就是说，只要属于视频编辑部分，就默认以 `Remotion` 为核心

对于 MCP，当前建议作为前端开发前准备项：

- 若存在可直接提升开发效率的 MCP 能力，可先接入再开始开发
- 其作用主要是辅助调试、排查和开发协作，不改变产品功能需求本身
